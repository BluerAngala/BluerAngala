# 从一句话到 TypeScript 架构——自生长 Agent 的 47 分钟诞生记

> 这是「自生长 Agent 开发日志」系列的第二篇文章。我会记录这个 AI 系统从零到一的完整生长过程，包括每个功能背后的思考、踩坑和妥协。如果你对 AI Agent 开发感兴趣，欢迎关注。

## 那天晚上 9 点 13 分

我打开终端，敲下第一行字。

不是什么宏大的架构设计，不是什么精心规划的 roadmap。就是一个想法：

**"如果 AI 能每隔几分钟醒来一次，自己思考一次，然后继续沉睡呢？"**

我写下了一个 100 行的 bash 脚本，一个 28 行的角色定义文件，一个 29 行的状态文件。就这些。心跳间隔 5 分钟，每天上限 288 次。

这就是 `heartbeat.sh`——整个 self-growing-agent 的第一个心跳。

## 从心跳到自生长

跑了几轮之后，我发现了一个问题：**纯内省很快就会枯竭。**

如果一个 AI 每 5 分钟醒来一次，只是对着自己的记忆发呆，那它很快就会反复说"当前没有需要思考的问题"。这不叫思考，这叫空转。

我需要给它两种模式：

- **有人问的时候**：听觉模式，回答问题，从对话中学习
- **没人问的时候**：自生长模式，自己决定做什么——整理记忆、探索问题、写文档、或者什么都不做

于是 `heartbeat.sh` 被替换成了 `run.sh`——266 行 bash，支持两种运行模式，三层记忆系统（工作记忆、长期记忆、成长日志），每 20 次心跳自动压缩一次记忆。

角色定义也从 28 行精简到一句话：**"你刚出生，对这个世界充满好奇。"**

我还写了一份 209 行的 `DESIGN.md`，画了系统架构图，定义了感知层、记忆层、思考层、行动层的交互方式。

一切看起来都在往好的方向走。

然后我意识到：**bash 不适合做这件事。**

## 10 分钟，推倒重来

21:33，我做了一个决定：用 TypeScript 重写整个系统。

不是因为 bash 有什么致命的 bug。而是当我想要做更复杂的事情——事件驱动、异步处理、模块化——bash 的局限性变得越来越明显。你不能用 bash 做 EventEmitter，你不能用 bash 做非阻塞 IO，你不能用 bash 做类型检查。

10 分钟后，项目结构变成了这样：

```
src/
├── main.ts              ← 中枢心跳
├── perception/
│   └── collect.ts       ← 感知层
├── memory/
│   ├── recall.ts        ← 读取记忆
│   ├── record.ts        ← 写入记忆
│   └── compress.ts      ← 压缩归档
├── thinking/
│   ├── think.ts         ← 调用 LLM 思考
│   └── role.md          ← 角色定义
├── storage/
│   └── save.ts          ← 储存层
├── action/
│   └── execute.ts       ← 行动层
└── utils/
    └── logger.ts        ← 日志工具
```

`run.sh` 被删除了。`DESIGN.md` 被删除了。`role.md` 被移到了 `src/thinking/` 下。

`package.json` 出现了。`tsconfig.json` 出现了。`npm start` 可以一键启动了。

## 修复，修复，修复

TypeScript 版本上线后，立刻暴露了一堆问题。

**第一个 bug：双重压缩。** `record.ts` 里有个自动压缩逻辑——每写入一次记忆就检查行数，超过 50 行就压缩。同时 `main.ts` 里也有个压缩逻辑——每 20 次心跳压缩一次。两个逻辑同时触发，记忆被压缩了两次。删掉了 `record.ts` 里的逻辑，只保留 `main.ts` 的。

**第二个 bug：pi 命令参数错误。** `pi --session` 应该是 `pi --name`。改了。

**第三个 bug：日期格式。** `toLocaleString` 在不同系统上返回的格式不一致，导致计数文件的文件名在换日时出问题。改成 `toISOString()`，直接取前 10 个字符当日期。

**第四个 bug：think.ts 的 SDK 调用失败。** 尝试用 pi 的 npm SDK 直接调用，但类型定义有问题，编译不过。回退到 CLI 方式——`execSync` 调用 `pi` 命令行。虽然不够优雅，但能跑。

**第五个 bug：重复日志。** `main.ts` 里有两行思考日志，删掉一行。

这些 bug 加起来修了 15 分钟。不算多，但每一个都是"跑起来才发现"的问题。

## 听觉：从文件到麦克风

TypeScript 架构稳定后，我开始做一件更大的事：**让 Agent 能真正听见声音。**

之前的"听觉"是假的——只是读取 `inbox.md` 文件的内容。真正的听觉需要：麦克风录音 → 静音检测 → 语音转文字。

我在 `collect.ts` 里加入了 ffmpeg 录音（每次录 10 秒），用 `volumedetect` 滤镜检测音量——低于 -50dB 就跳过，说明没人说话。有声音的话，调用硅基流动的 STT API 转成文字。

角色定义也改了。不再是"等待指令"，而是：

> **"你不是在等待指令。你在主动探索。"**

Agent 可以自己决定什么时候听、什么时候不听——通过 `[hear: on]` 和 `[hear: off]` 指令控制麦克风开关。如果你在思考一个复杂问题，可以关掉听觉集中注意力；如果你想了解周围发生了什么，就开启听觉。

## 进程拆分：麦克风不能抢

把听觉逻辑放在 `collect.ts` 里有个问题：**麦克风是独占资源。** 主进程每 3 分钟心跳一次，每次心跳都要录音 10 秒，这意味着大部分时间麦克风是空闲的，但又被主进程锁着。

解决方案：把听觉拆成独立进程。

`hearing.ts` 作为一个单独的进程运行（`npm run hear`），持续录音。有声音就写入 `inbox.md`。主进程的心跳从 `inbox.md` 读取内容，决定要不要继续听。

两个进程通过文件通信，互不干扰。

## 47 分钟后的样子

从 21:13 到 22:00，47 分钟，12 个 commit。self-growing-agent 从一个 bash 心跳脚本变成了一个 TypeScript 事件驱动系统：

- **三层记忆**：工作记忆 + 长期记忆 + 成长日志
- **感知层**：文件变化检测 + 麦克风录音 + 语音转文字
- **思考层**：调用 LLM 进行自主思考
- **行动层**：可扩展的工具调用系统
- **听觉控制**：Agent 自己决定听还是不听
- **进程隔离**：主进程和听觉进程独立运行

它还不完美。静音阈值是硬编码的，STT 依赖网络，TTS 还没加上。但框架已经搭好了。

## 这段经历教会了我什么

**先跑起来，再优化。** 心跳脚本只有 100 行，但它立刻给了我反馈——纯内省会枯竭。这个反馈直接改变了整个项目的方向。

**bash 有极限。** 当你需要事件驱动、异步处理、模块化的时候，bash 的抽象能力就不够了。TypeScript 不是更"高级"，而是更适合这个场景。

**bug 是免费的架构课。** 双重压缩 bug 教会了我"单一职责原则"；进程拆分教会了我"关注点分离"。每个 bug 背后都是一次架构认知的升级。

**Agent 应该有自主权。** 让它自己决定什么时候听、什么时候不听，比硬编码一个"始终听"的逻辑要聪明得多。
