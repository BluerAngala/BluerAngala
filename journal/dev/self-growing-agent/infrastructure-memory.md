# 给 AI Agent 造一套"五感"和"记忆检索"——从能思考到能自主探索

> 这是「自生长 Agent 开发日志」系列的第四篇。上一篇（[第一篇](./2026-06-26-hearing-journey.md)）讲了听觉系统，这篇讲基础设施和记忆系统。我记录了 Agent 从"能想"到"能自主探索"的关键跨越。

## 从一个 Bug 说起

我的自生长 Agent 跑了一段时间后，我发现了一个诡异的行为：**它在反复读同一个文件。**

每次心跳触发思考，它都会去读 `growth.md`，看完之后写一段新内容进去。下一次心跳，它又去读——因为内容变了，所以又写一段。再下一次，又读。循环往复，`growth.md` 从几百字疯长到几千字，全是同一种思考模式的重复。

Agent 陷入了自我对话的死循环。

这不是一个普通的 bug。它暴露了一个更根本的问题：**这个系统还不具备"时间感知"。** 它不知道自己 5 分钟前刚读过这个文件，不知道内容没有外部变化，不知道该把注意力放到新的东西上。

在人类的世界里，我们靠记忆来做判断——"这个我看过了"、"那个还没碰"。但在 Agent 的世界里，每次心跳都是一次全新的"醒来"，它没有时间线的概念。

我需要给它造一套基础设施来解决这些问题。

## 路线图：从婴儿到成年

在动手之前，我先画了一张路线图。把 Agent 的成长分成 6 个阶段，模拟人从婴儿到成年的发育过程：

```
婴儿期 → 幼儿期 → 儿童期 → 少年期 → 青年期 → 成年期
能听    能记      能做      能探索    有想法    能自生长
```

这个路线图不是装饰。每个阶段有明确的能力定义和限制，阶段之间有依赖关系——上一个阶段不稳固，下一个阶段就会出问题。

当时，婴儿期和幼儿期已经完成了（能感知、能读文件、能写文件）。儿童期正在进行中。但那个"反复读同一个文件"的 bug 说明，从儿童期到少年期的跨越遇到了真正的障碍。

我决定先修基础设施。

## 多感官感知系统

第一步是重构感知层。

原来的感知很简陋：检查文件变化、读 inbox 消息。相当于 Agent 只有"眼睛"——能看到文件变了，但其他什么都不知道。

我给它加了四层感知：

**时间感** — Agent 现在知道自己什么时候醒来，现在是几点、什么时段（深夜/早晨/上午/中午/下午/晚上）。人类的时间感来自生物钟，Agent 的来自系统时钟——但作用是一样的：知道"现在是什么时候"，才能决定该做什么。

**空间感** — Agent 知道自己住在哪里（项目目录结构），data/ 里有哪些文件、每个文件多大。就像一个人走进自己的房间，扫一眼就知道东西在不在、位置对不对。

**状态感** — Agent 知道自己的记忆有多大（working.md、growth.md、long-term.md 分别多大），日志累积了多少。这是"自我感知"——知道自己的内部状态，才能判断是否需要整理记忆、是否该压缩旧数据。

**变化感** — 这是之前就有的，但做了大幅改进。从只监控 `data/` 和 `src/` 两个目录，扩展到整个项目。扫描整个目录树，对比快照，精确报告"哪些文件新增了、哪些修改了、哪些删除了"。还排除了 logs、node_modules 这些噪音目录。

这五层感知组装起来，Agent 每次心跳收到的不再是一句"有文件变化"，而是：

```
【听到】用户在 inbox.md 留了消息
【变化】src/thinking/think.ts 被修改了
【时间】2026/6/26 08:20（早晨）
【空间】根目录: data/logs/src | 文件: package.json, README.md...
【状态】记忆: 工作2.1KB, 长期0.5KB, 成长1.1KB | 日志3.2KB
```

有了这套感知系统，Agent 第一次能"立体地"理解自己所处的环境。

## 记忆去重：读取历史

回到开头的 bug。Agent 反复读同一个文件，本质原因是它没有"读取历史"的记忆。

我新建了 `readHistory.ts` 模块，记录每次文件读取的路径、时间、大小。Agent 读文件前先查历史：如果 5 分钟内读过同一个文件，就直接告诉它"你刚看过这个，内容没变"，不再重复读取。

实现很朴素：一个 JSON 文件存最近 50 条记录，超过就截断。但效果立竿见影——Agent 不再陷入自我对话循环，`growth.md` 从几千字的垃圾日志，变成了简洁的成长记录。

更妙的是，读取历史会随 prompt 一起发给模型。Agent 在思考时能看到自己最近读过哪些文件，自然地把注意力放到还没读过的东西上。这就像人类走进图书馆，不会反复翻同一本书——因为我们记得自己看过了。

## bash 执行：白名单策略

光能读写文件还不够。Agent 还需要更灵活的文件探索能力——比如 `find` 查找文件、`grep` 搜索内容、`du` 看目录大小。

但直接给 Agent 开放 bash 是危险的。一个不谨慎的 `rm -rf` 就能毁掉整个项目。

我采用了白名单策略：只允许 13 个只读命令（ls, cat, find, grep, wc, head, tail, echo, date, pwd, file, du, stat）。Agent 调用 `run_command` 时，先检查命令的第一个词是否在白名单里，不在就拒绝。

路径安全也做了加固：所有文件路径都会经过 `resolve` + `BASE` 前缀检查，防止 Agent 读写项目目录之外的文件。

这两层防护看起来简单，但对 Agent 的自主探索至关重要——它现在能安全地使用 shell 工具来探索世界，同时不会搞出破坏。

## 记忆搜索

Agent 有了读取历史之后，不再重复读文件了。但新的问题出现了：**记忆文件越来越大，Agent 需要的不是"读整个文件"，而是"找到相关内容"。**

就像人类不会把整本笔记从头翻到尾——我们会翻目录、搜关键词、跳到需要的部分。

我新建了 `search.ts` 模块，实现了关键词搜索功能：Agent 给一个关键词，系统在 working.md、growth.md、long-term.md 三个记忆文件中搜索，返回匹配的行和上下文。

这和 bash 的 `grep` 不同。`grep` 是原始文本搜索，而 `search_memory` 是面向记忆的搜索——它知道记忆文件的结构，能给出更有意义的结果（文件名、匹配位置、前后文）。

Agent 现在有五种获取信息的方式：`read_file`（精确读取）、`write_file`（写入）、`list_dir`（列目录）、`run_command`（shell 探索）、`search_memory`（关键词搜索）。这五种工具组合在一起，构成了 Agent 探索世界的基础能力。

## 工程化：Biome + 配置集中

在功能开发的间隙，我做了一次工程化清理。

之前用 ESLint + Prettier，两个工具的配置很分散。换成了 Biome——一个集 lint 和 format 于一体的工具。配置文件从两个减到一个，规则更清晰。跑了 `biome check`，0 warnings，代码干净。

更重要的是 `config.ts`：把散落在各处的配置项（API key、模型名、心跳间隔、每日上限、白名单等）集中到一个文件里。以前改一个参数要在三四个文件里找，现在只需要改一个地方。

这种清理看似不起眼，但对后续开发影响很大。项目变复杂后，"配置在哪里"比"配置是什么"更重要。

## 这个阶段教会我的事

回顾这 12 个 commit，核心教训是三个：

**1. 自主性需要基础设施。** Agent 不能只是"能思考"——它需要时间感知来避免重复，需要空间感知来理解环境，需要记忆搜索来高效检索。没有这些基础设施，所谓的"自主探索"只是随机游走。

**2. 安全和自由是同一枚硬币的两面。** 给 Agent 开放 bash，同时用白名单限制它。给它文件读写权限，同时做路径校验。没有安全边界的自由是混乱，没有自由的安全是禁锢。

**3. 路线图比代码更重要。** 在修 bug 和做新功能的冲动中，停下来画一张路线图——定义每个阶段的能力和限制——让我避免了"功能蔓延"。每次想加新东西，先问：这属于哪个阶段？上一个阶段稳固了吗？

## 下一步

Agent 现在能感知环境、管理记忆、安全地执行命令。阶段四（少年期）已经完成——它开始主动探索项目结构、建立心智模型。

接下来是阶段五（青年期）：让 Agent 有自己的想法，而不只是机械地执行工具调用。它需要学会写长文、整理记忆、向主人汇报。更重要的是，它需要学会问问题——不是被动地等待指令，而是主动地产生好奇心。

这条路还很长。但至少，基础设施已经到位了。
