# 踩坑与改进——从听觉 v1 到语音对话的「边修边长」

> 这是「自生长 Agent 开发日志」系列的第五篇文章。我会记录这个 AI 系统从零到一的完整生长过程，包括每个功能背后的思考、踩坑和妥协。如果你对 AI Agent 开发感兴趣，欢迎关注。

## 为什么这篇文章存在

上一篇写了听觉系统从 v1 到 v7 的 20 次迭代，但其实有几组 commits 被跳过了——它们要么太早（v1 架构的听觉实验），要么太新（v7 之后的体验优化）。这些被跳过的 commits 反而藏着最有意思的教训。

这篇文章补上这些空白。

## 第一幕：v1 时代的听觉尝试（08:43 - 08:54）

在 v2 事件驱动架构出现之前，我做了第一次听觉尝试。这 4 个 commits 发生在同一天上午 8 点半左右，只用了 11 分钟。

### ffmpeg + 云端 STT（f439145）

第一版方案：用 ffmpeg 录 10 秒音频，发到硅基流动的 SenseVoice STT API 转成文字，写入 `inbox.md`。

```bash
npm run hear  # 独立进程，手动启动
```

听起来很合理，但实际跑起来：
- 每 10 秒才处理一次，延迟太高
- 云端 API 有网络延迟，加上录音时间，说完话要等十几秒才有反应
- 需要手动启动两个进程（听觉 + 主进程）

**但至少证明了一件事：麦克风 → 文字 → 文件 → agent 这条管道是通的。**

### 本地 Apple Speech（ba4ddb5）

云端太慢了。3 分钟后，我写了第二个版本：用 Swift 调用 macOS 的 Apple Speech 框架，零 API 调用，完全本地。

```swift
import Speech
let recognizer = SFSpeechRecognizer(locale: Locale(identifier: "zh-CN"))
```

两种模式并存：
- `npm run hear:local` — macOS 原生，免费，离线可用
- `npm run hear:cloud` — 硅基流动 STT，需要 API key

Apple Speech 的优势是零延迟、零成本，但它有自己的问题：识别质量不如专业 STT 模型，偶尔会乱认字。不过对于"验证管道能不能跑通"这个目标，够用了。

### 管道验证（7174948）

Apple Speech 版本跑通了完整的管道：

```
麦克风 → Apple Speech STT → inbox.md → 感知层 → 思考层 → 回应
```

我对着麦克风说"你好，我是陈恒律师"，agent 收到了，读了 ROADMAP 理解自己处于阶段四，写了 389 字回应。**这是第一次真正意义上的语音对话。**

虽然管道只跑通了一次，但它证明了一个关键假设：**不需要复杂的架构，把录音结果扔进 inbox.md，现有的感知-思考-行动管道就能处理。**

### 共享路径 + 自言自语防护（63f1f6f）

管道通了，但留下了几个严重问题。

**路径地狱**：每个模块自己算 `../..` 来定位 `data/` 目录。`hearing.ts` 算一遍，`collect.ts` 算一遍，`memory/` 下的每个文件又算一遍。一旦有一个算错了，整个系统就找不到文件。

解决方案：创建 `paths.ts`，所有模块统一引用：

```typescript
// paths.ts — 一处定义，到处引用
const dir = import.meta.dirname ?? process.cwd();
export const BASE = resolve(join(dir, ".."));
export const DATA = join(BASE, "data");
export const INBOX = join(DATA, "inbox.md");
```

**自言自语循环**：agent 说话时，麦克风会录到自己的声音。agent 听到"声音"后又去思考、又去回复，形成死循环。

解决方案：`write_file` 工具直接禁止写入 `inbox.md`——这是感知层的领地，agent 不能自己往里塞消息。

```typescript
// perception/collect.ts
if (filePath.endsWith('inbox.md')) {
  return { success: false, error: '禁止写入 inbox.md（防止自言自语）' };
}
```

同时清理了 biome lint 到 0 errors。17 个文件被修改，净减少 14 行代码。

**这 4 个 commits 虽然只花了 11 分钟，但奠定了后来 v2 架构的听觉基础。** 没有 v1 的这些实验，就不会知道"管道通了但路径和自言自语是大问题"——而这些问题在 v2 重构中被系统性地解决了。

## 第二幕：语音对话验证通过（11:00）

时间来到 v2 架构的听觉系统经过 20 次迭代之后。v7 听觉已经稳定了，现在需要验证一件事：**这东西真的能用来对话吗？**

1aa72b3 这个 commit 记录了第一次成功的连续对话：

| 我说了什么 | Agent 回应时间 |
|------------|----------------|
| "啊，你是" | 4.6 秒 |
| "想看看你还活着吗" | 4.0 秒 |

两个关键观察：
1. **回声被捕获但自然处理了** —— agent 自己说话时，麦克风确实录到了回声，但系统没有陷入死循环。v1 时代的自言自语防护在这里生效了。
2. **心跳静默正常** —— 对话期间心跳没有打断，思考层正确地把语音输入当作用户消息处理。

4-6 秒的响应延迟，对于一个"说话 → 等待 → 回复"的对话来说，已经可以接受了。虽然比不上人类对话的即时性，但至少不再是"说完话要等十几秒"。

## 第三幕：确认音和打断检测——以及为什么又删掉了（11:01 - 11:02）

验证通过后，我开始优化对话体验。

### 确认音（9cc7027）

人和人对话时，你说完一句话，对方会"嗯"一声表示在听。agent 也应该这样。

```typescript
// 收到消息立刻播放确认音
export function acknowledge(): void {
  execSync(`say -v Ting-Ting -r 200 "嗯"`, { timeout: 3000 });
}
```

这个改动很简单，但体验提升巨大。用户说完话后，1 秒内就能听到"嗯"，知道 agent 听到了。然后才是完整的回复。

### 打断检测（9cc7027）

更野心勃勃的尝试：agent 说话时，持续监听麦克风（每 500ms 检查一次）。检测到用户在说话 → 立刻停止朗读。

实现方式是 `spawn` 一个后台 `say` 进程（而不是 `execSync`），同时用定时器检查麦克风能量：

```typescript
// 后台播放，同时监听打断
const proc = spawn("say", ["-v", VOICE, "-r", RATE, clean]);
proc.on("close", () => { /* 播放结束 */ });

// 每 500ms 检查麦克风
const checkInterval = setInterval(() => {
  if (isUserSpeaking()) {
    proc.kill();  // 停止朗读
    onInterrupted?.();
  }
}, 500);
```

### 23 秒后，全部删掉（5aebd9d）

打断检测上线仅 23 秒，我就把它删了。

原因：**打断检测需要持续监听麦克风，而听觉系统也在监听麦克风。两个进程抢同一个音频设备，直接冲突。**

macOS 的麦克风是独占资源——同一时间只能有一个进程在用。听觉进程在录音，打断检测也在录音，结果就是：两边都拿不到完整的音频流，语音识别质量暴跌。

删掉打断检测后，代码从 56 行缩减到 12 行。`spawn` 改回 `execSync`，后台播放改回同步播放。

**教训：好的想法如果和系统约束冲突，就要果断放弃。** 打断检测在概念上很美好，但在 macOS 的音频子系统面前，它和听觉系统不可能共存。最终靠"轮次"解决——用户说完等 agent 回复完，再继续说，就是正常的对话节奏。

## 7 个 commits 的共同主题

回头看这 7 个 commits，它们都在回答同一个问题：**怎么让 agent 和人自然地对话？**

| 问题 | 尝试 | 结果 |
|------|------|------|
| 怎么让 agent 听到我说话？ | ffmpeg + 云端 STT | 能用，但太慢 |
| 能不能更快？ | Apple Speech 本地识别 | 能用，零延迟 |
| 管道通了吗？ | 端到端验证 | 通了，但有路径和循环问题 |
| 路径到处算，怎么统一？ | paths.ts 共享路径 | 解决 |
| agent 会不会自言自语？ | write_file 禁写 inbox.md | 解决 |
| 用户说完话 agent 知道吗？ | 确认音"嗯" | 体验好 |
| 能不能打断 agent 说话？ | 打断检测 | 和麦克风冲突，删了 |

## 从中学到的 3 件事

### 1. 先跑通管道，再优化架构

v1 的听觉虽然粗糙（ffmpeg 录 10 秒、云端转文字、写文件），但它证明了管道是通的。这个验证让后面的 v2 事件驱动架构有了明确的方向——不是从零开始，而是把已验证的管道用更好的架构重新实现。

### 2. 共享状态要用共享机制管理

路径计算这种"每个模块各算各的"的模式，是 bug 的温床。`paths.ts` 只有 11 行代码，但它消除了一个系统性的故障源。**当多个模块需要同一个值时，不要让每个模块自己算——给它一个共享的来源。**

### 3. 硬件约束是最终裁判

打断检测在软件层面完全可行——后台播放 + 定时器监听。但 macOS 的麦克风独占模型让这个方案在物理上不可行。**软件设计再优雅，也跑不过硬件的硬约束。** 遇到这种情况，不要试图绕过去，接受它，找替代方案。
